## P1. アプリ開発者「ミナ」

- **役割**: バックエンド/フルスタック開発者。ローカルで複数コンテナ（API, DB, cache 等）を回しながら開発。
- **ゴール**: 動かしているコンテナの状態を素早く把握し、落ちたサービスのログを即座に確認、必要なら入って調べたい。
- **ペインポイント**:
  - `docker ps` → ID コピー → `docker logs -f` の往復が煩雑。
  - Docker Desktop は重く、起動も遅い。
  - どのコンテナが落ちているか一目で分かりにくい。
- **行動特性**: キーボード中心。vim ユーザー。ツールの見た目とリズムを重視。
- **重視する価値**: 速さ、状態の色分け、ログ追尾、exec への即アクセス。
- **関連ストーリー**: US-1, US-2, US-3, US-5, US-6, US-10, US-11 / v2: US-15, US-16, US-19

## P2. DevOps / SRE「ケンジ」

- **役割**: 本番・ステージング環境を SSH 越しに調査する運用担当。
- **ゴール**: リモートホスト上で、リソースを食っているコンテナや異常なログを安全に特定し、最小限の操作で対処したい。
- **ペインポイント**:
  - GUI が使えない環境が多い。CLI 反復は調査時に遅い。
  - 既存 TUI がコンテナ内/SSH 越しでログや統計を表示できないことがある。
  - 削除など破壊的操作の誤爆が怖い。
- **行動特性**: 慎重。安全性と信頼性を最優先。多数コンテナを扱う。
- **重視する価値**: SSH/コンテナ内でも動く、統計表示、確認ダイアログ、フィルタ、軽快さ。
- **関連ストーリー**: US-2, US-4, US-7, US-8, US-9, US-12, US-13 / v2: US-17, US-18, US-20, US-21

## P3. ホームラボ運用者「アヤ」

- **役割**: 自宅サーバ / Raspberry Pi で趣味のコンテナ群を運用。
- **ゴール**: 設定不要で起動し、コンテナを眺めて時々再起動・更新する。軽くて気持ちよく使えるツールがほしい。
- **ペインポイント**:
  - 重いツールは非力なマシンで動かない。
  - 機能過多で複雑なものは覚えるのが面倒。
- **行動特性**: 低頻度利用。直感的に使えることが重要。見た目の楽しさも動機。
- **重視する価値**: ゼロコンフィグ、軽さ、シンプルさ、カラフルさ、discoverability。
- **関連ストーリー**: US-1, US-2, US-6, US-11, US-14 / v2: US-15（自宅の compose スタックをまとめて操作）, US-20
